# Architecture Principles: Flutter Forge 🏛️

This document describes the architectural decisions, patterns, and principles embedded within the code generated by Flutter Forge.

---

## 🧭 1. Core Architectural Separation

We adhere to a strict philosophy:
> **"Flutter Forge owns only the application architecture; Flutter owns the platform."**

This separation ensures that native runners (Android, iOS, Web, Desktop), platform channels, and system hardware bindings remain within the domain of standard Flutter tooling. Forge focuses purely on organizing Dart code, configuring dependencies, structuring data pipelines, and streamlining UI-to-service communication.

```
┌────────────────────────────────────────────────────────┐
│                   FLUTTER FORGE                        │
│             (Application Architecture)                 │
│                                                        │
│  ┌──────────────┐   ┌──────────────┐   ┌────────────┐  │
│  │ Presentation │   │    Domain    │   │    Data    │  │
│  └──────────────┘   └──────────────┘   └────────────┘  │
└──────────────────────────┬─────────────────────────────┘
                           │ Consumes / Executes
┌──────────────────────────▼─────────────────────────────┐
│                    FLUTTER ENGINE                      │
│                   (Platform Code)                      │
│                                                        │
│  ┌──────────────┐   ┌──────────────┐   ┌────────────┐  │
│  │   Android    │   │     iOS      │   │    Web     │  │
│  └──────────────┘   └──────────────┘   └────────────┘  │
└────────────────────────────────────────────────────────┘
```

## 📂 2. Feature-First Clean Architecture

Flutter Forge structures applications around **features** rather than technical layers. This minimizes file-hunting and cognitive overhead by keeping all logical blocks of a single feature close to each other.

Inside each feature, code is separated into three distinct Clean Architecture layers:

```
                  ┌──────────────────────┐
                  │  Presentation Layer  │
                  │  (Screens & Widgets) │
                  └──────────┬───────────┘
                             │
                             │ Uses (Entities/Use Cases)
                             ▼
                  ┌──────────────────────┐
                  │     Domain Layer     │
                  │ (Entities, Use Cases,│
                  │  Repository Contract)│
                  └──────────▲───────────┘
                             │
                             │ Implements (Contracts)
                             │
                  ┌──────────┴───────────┐
                  │      Data Layer      │
                  │ (Models, DataSources,│
                  │  Repository Impls)   │
                  └──────────────────────┘
```

### A. The Domain Layer (Core Business Logic)
The domain layer is the most isolated layer in the application. It has **zero dependencies** on external data sources, networks, database layers, or UI packages.
*   **Entities**: Pure Dart objects representing core business concepts.
*   **Use Cases**: Encapsulated orchestrators of user actions (e.g. `LoginUseCase`).
*   **Repository Interfaces**: Contracts defining methods for retrieving/sending data. The data layer implements these contracts, ensuring dependency inversion.

### B. The Data Layer (Infrastructure & Persistence)
The data layer is responsible for retrieving, caching, parsing, and storing data.
*   **Models**: JSON serialization wrappers extending Domain Entities.
*   **Data Sources**: Remote API wrappers (Dio calls) or local storage handlers (SQLite, SharedPreferences).
*   **Repository Implementations**: Implementations of Domain Repository interfaces. Coordinates data retrieval logic (e.g., check local cache; if expired, fetch from remote and write to local cache).

### C. The Presentation Layer (User Interface)
The presentation layer manages user interaction and renders data.
*   **Screens**: Layout-level widgets representing complete pages.
*   **Widgets**: Highly reusable components focused on rendering parts of the screen.
*   **State Controllers**: State management classes (e.g., BLoC, Cubit, Riverpod Notifier) that receive inputs from the UI, execute domain use cases, and emit state updates back to the UI.

## 🛡️ 3. Standardized Exception & Error Mapping

Instead of catching raw network errors (e.g., `DioException`) at the UI layer, Flutter Forge implements a centralized boundary mapping pattern.

### The Error Pipeline:
1.  **Network Transport**: A network call encounters a failure (e.g., a HTTP 401 unauthorized status code).
2.  **Interceptor Interception**: The custom Dio interceptor captures the raw failure.
3.  **Exception Mapping**: The repository catches the network layer error and maps it using an `ExceptionMapper` to a standardized domain-specific `AppException` (e.g., `UnauthorizedException`).
4.  **UI Propagation**: The UI (via State Controllers) handles the typed `AppException` cleanly, displaying localized error views or warnings without exposing raw server details.

```
[Raw HTTP / Dio Error] ──► [ExceptionMapper] ──► [AppException] ──► [State Controller] ──► [Friendly UI State]
```

## 💉 4. Dependency Injection & Service Locator

Flutter Forge utilizes `get_it` as a Service Locator to resolve dependencies cleanly.
*   **Decoupled Registration**: Rather than one monolithic global file, each feature file declares its own dependencies. These are registered using a feature initialization block called by the global configuration loader.
*   **Scope Isolation**: Ensures services and repositories are instantiated as singletons when appropriate, and presentation controllers are unregistered when screens are popped to preserve memory.
