# Folder Structure: Flutter Forge 📂

This document maps out the comprehensive directory structure generated by Flutter Forge, illustrating how it scales from a small application to an enterprise-grade codebase.

---

## 🌳 1. The Directory Tree

Here is the folder hierarchy created when you initialize a project using `forge init` and scaffold features using `forge add feature`:

```
my_app/
├── .env.dev                        # Environment variables for local development
├── .env.staging                    # Environment variables for staging builds
├── .env.prod                       # Environment variables for production releases
├── forge.yaml                      # Flutter Forge configuration file
├── pubspec.yaml                    # Dart package configuration
└── lib/
    ├── main_dev.dart               # Entry point for development flavor
    ├── main_staging.dart           # Entry point for staging flavor
    ├── main_prod.dart              # Entry point for production flavor
    │
    ├── core/                       # Shared logic across multiple features
    │   ├── di/
    │   │   └── injection.dart      # Global Dependency Injection (Service Locator) setup
    │   ├── exceptions/
    │   │   ├── app_exception.dart  # Custom domain exception classes
    │   │   └── mapper.dart         # Maps Dio/HTTP errors to AppExceptions
    │   ├── network/
    │   │   ├── api_client.dart     # Dio HTTP client wrapper class
    │   │   ├── endpoints.dart      # Centralized API endpoint paths
    │   │   └── interceptors/       # HTTP interceptors (auth, logger, tracing)
    │   ├── router/
    │   │   ├── app_router.dart     # Central go_router definition
    │   │   └── routes.dart         # Core application route path strings
    │   └── theme/
    │       └── app_theme.dart      # Standard dark/light application theme specifications
    │
    └── features/                   # Feature modules
        └── auth/                   # Example: Authentication feature
            ├── data/
            │   ├── datasources/    # Remote API endpoints & local storage calls
            │   │   └── auth_remote_datasource.dart
            │   ├── models/         # JSON parsing models mapping server payload
            │   │   └── auth_model.dart
            │   └── repositories/   # Implementation of the domain repository contract
            │       └── auth_repository_impl.dart
            ├── domain/
            │   ├── entities/       # Pure Dart domain objects
            │   │   └── user_entity.dart
            │   ├── repositories/   # Abstract repository interface contracts
            │   │   └── auth_repository.dart
            │   └── usecases/       # Unique business actions
            │       └── login_usecase.dart
            └── presentation/
                ├── controllers/    # State management controller (BLoC/Notifier/Cubit)
                │   └── auth_bloc.dart
                ├── screens/        # Main route endpoints/widgets
                │   └── login_screen.dart
                └── widgets/        # Internal UI reusable blocks (buttons, fields)
                    └── auth_input_field.dart
```

## 🔍 2. Folder Explanations

### Root Files
*   **`.env.*`**: Text files containing environment-specific configurations (e.g. `BASE_URL=https://api.dev.example.com`, `API_KEY=xyz`). Loaded dynamically into memory at build time.
*   **`forge.yaml`**: The configuration file that records architectural choices (state management, router package name) so subsequent CLI runs remain consistent.

### Main App Entry Points
*   **`lib/main_*.dart`**: Separate entry points corresponding to each configuration flavor. Each script initializes environment configs before starting the execution of the main Flutter app widget (e.g., `main_dev.dart` loads `.env.dev` first).

### `lib/core/` (Cross-Cutting Concerns)
*   **`di/`**: Central location for registering global utilities like networking clients, configuration configurations, local database configurations, and theme registries.
*   **`exceptions/`**: Contains error management definitions. Avoids catching generic raw exceptions inside screens.
*   **`network/`**: Defines HTTP routing client settings, response mapping definitions, request headers management, and security setups (like SSL Pinning or OAuth token rotation interceptors).
*   **`router/`**: Defines the main routing table structure, authorization guards, path bindings, and animation transitions.
*   **`theme/`**: Implements light and dark styling configurations, text hierarchies, standard colors, and screen border shapes.

### `lib/features/` (Feature Modules)
*   **`data/`**: Deals with network configurations, API integrations, and JSON transformations. Only domain layer adapters are exposed.
*   **`domain/`**: Houses independent logic. Excludes third-party API dependencies or UI controls.
*   **`presentation/`**: Handles user interfaces, state management emissions, theme rendering, and platform screen navigation.
